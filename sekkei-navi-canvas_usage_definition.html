1. アプリの方向性

このアプリは、いきなり実装へ進むのではなく、「何を作るのか」「なぜ作るのか」「第三者がどう理解するのか」を先に固めるためのアプリである。

解決する課題

開発初期に要求が曖昧なまま進み、チーム内で目的・範囲・優先順位がずれる問題を減らす。

中核コンセプト

「要求」を「要件」に変換し、さらに図で説明できる設計へ落とし込む。

成果物

目的、利用者、要件、設計図、検証基準を含むHTML形式の使用定義書を出力する。

作るもの

要求整理 要件定義 設計図作成 画面・機能対応表 レビュー支援 HTML出力

初期版で作らないもの

対象外 理由
実アプリ本体の自動生成 本アプリの目的は、実装前の共通理解と設計の明確化であるため。
本格的なUMLエディタ 初期版では、精密な作図よりも「図で説明できる」ことを優先するため。
チームチャット コメントとレビュー記録に絞り、コミュニケーション機能を肥大化させないため。
AIによる完全自動設計 設計責任は利用者に残し、アプリは思考整理と抜け漏れ確認を支援するため。

設計思想: 機能を増やすよりも、第三者が「目的、範囲、構造、検証方法」を説明できる状態を作る。

2. 利用者と境界

対象利用者は、情報系の学習者、学生チーム、小規模な開発チーム、要求分析や設計図の練習をしたい人である。

利用者 目的 主な操作
設計者 要求を要件へ変換し、図と説明を作成する。 プロジェクト作成、要求登録、図作成、HTML出力
レビュー担当者 設計の抜け漏れ、矛盾、理解しにくい箇所を指摘する。 閲覧、コメント、レビュー完了記録
チームメンバー 何を作るか、どこまで作るかを共有する。 使用定義書の閲覧、要件確認
教員・評価者 要求、設計図、検証基準が論理的に接続されているか確認する。 成果物閲覧、評価コメント

コンテキスト図

システム境界は「設計ナビ Canvas」本体。外側に利用者、レビュー担当者、出力HTML、データ保存先を置く。

3. 要求と要件

要求は利用者の言葉であり、要件は実装・検証できる条件である。設計ナビ Canvas は、この変換過程を画面上で支援する。

曖昧な要求 定義し直した要件 測定方法
チームで分かりやすい設計書を作りたい。 目的、利用者、機能要件、非機能要件、主要図、V&Vを1つのHTMLに出力できる。 第三者が10分以内に目的と主要機能を説明できるか確認する。
図を使って説明したい。 コンテキスト図、ユースケース図、操作フロー図、ER図、状態遷移図、構成図を作成できる。 各要件に関連する図が1つ以上紐づいているか確認する。
抜け漏れを減らしたい。 要求、要件、画面、図、テストの対応表を表示し、未対応項目を警告する。 未対応要件と参照切れを検出できるかテストする。
レビューしやすくしたい。 各要件・図・画面にコメントと対応状況を記録できる。 レビューコメントの登録、完了、再確認ができるか確認する。

機能要件

ID 要件 優先度 関連図
FR-01 プロジェクトの目的、利用者、前提条件、対象外を登録できる。 高 コンテキスト図、ユースケース図
FR-02 要求を機能要求、非機能要求、制約条件に分類できる。 高 DFD、要求対応表
FR-03 要件、画面、図、テストを相互リンクできる。 高 トレーサビリティ表
FR-04 主要な設計図をテンプレートから作成できる。 高 各種設計図
FR-05 コメント、レビュー結果、変更履歴を記録できる。 中 状態遷移図、権限表
FR-06 使用定義書をHTMLとして出力できる。 高 シーケンス図、コンポーネント図

非機能要件

分類 要件 合格基準
使用性 非エンジニアでも全体像から詳細へ自然に読み進められる。 初回利用者がサンプル使用定義書を読み、主要目的を説明できる。
性能 中規模プロジェクトでも編集・表示が遅くならない。 100要件、20図、100コメントのプロジェクトを3秒以内に表示する。
信頼性 自動保存、復元、変更履歴により上書き事故を防ぐ。 通信断後も直近1分以内の編集内容を復元できる。
監査性 誰が、いつ、どの要件や図を変更したか追跡できる。 変更履歴から変更者、日時、対象、差分を確認できる。
移植性 主要ブラウザでHTML使用定義書を閲覧できる。 Chrome、Edge、Safari、Firefoxで表示崩れがない。

4. AIチャットボット・会社リスト連携

設計ナビ Canvas は、AIチャットボットと連携して、会社リストを取得・整理し、設計書や比較表に取り込める。ただし、AIに会社一覧を推測させるのではなく、信頼できるデータ源から取得した一覧をAIが整形・分類・説明する設計にする。

想定する利用場面

利用場面 できること 注意点
就活・企業研究 業界、地域、職種、上場区分などで会社リストを取得し、比較表を作る。 募集状況や採用情報は変わるため、取得日と出典を必ず残す。
営業・市場調査 対象業界の会社を一括取得し、規模、所在地、事業内容で分類する。 商用データベースやWebサイトの利用規約を確認する。
設計課題の題材収集 比較対象企業を集め、要求分析やペルソナ作成の材料にする。 AIの要約は参考情報とし、正式な会社情報は出典で確認する。

AIとデータ取得の責務分担

担当 役割 禁止・制限
データ取得機能 API、CSV、社内DB、許可されたWebデータから会社一覧を取得する。 利用規約に反するスクレイピングや、出典不明データの混入を避ける。
AIチャットボット 自然文の条件を検索条件へ変換し、会社リストを分類・要約・比較する。 存在確認できない会社を事実として追加しない。
設計ナビ Canvas 取得条件、出典、取得日時、加工結果、使用目的を使用定義書に記録する。 出典や更新日がない一覧を正式成果物として扱わない。

機能要件の追加

ID 要件 合格基準
FR-AI-01 利用者が「東京都のIT企業を一覧化したい」のような自然文で会社検索条件を入力できる。 AIが業界、地域、規模、用途などの検索条件に分解して確認画面へ表示する。
FR-AI-02 会社リストをAPI、CSV、管理者登録データのいずれかから取り込める。 会社名、所在地、業種、URL、出典、取得日時を保持できる。
FR-AI-03 取り込んだ会社一覧を重複排除し、条件に合わない項目を除外候補として表示できる。 同一URLまたは同一法人番号の重複を検出できる。
FR-AI-04 会社リストを設計書内の表、比較図、調査対象リストとして挿入できる。 使用定義書に取得条件、件数、出典、更新日が表示される。

会社リスト取得フロー

重要: 「丸ごと持ってくる」は可能だが、対象範囲、件数上限、取得元、利用許諾、更新日を明記する。AIの回答だけを会社マスタとして扱わない。

8. 品質とV&V

このアプリの品質は「動くこと」だけでは測れない。第三者が読んで理解できる使用定義書を作れることを品質基準に含める。

V字モデル

受け入れ条件

  • ✓新規プロジェクトを作成し、目的・利用者・対象外を登録できる。
  • ✓要求を要件へ変換し、優先度と合格基準を設定できる。
  • ✓要件、画面、図、テストを相互に関連づけられる。
  • ✓未対応要件、リンク切れ、テスト未定義を検出できる。
  • ✓HTML使用定義書を出力し、主要ブラウザで崩れなく閲覧できる。
  • ✓第三者が10分以内にアプリの目的、主要機能、作らないことを説明できる。

PERT図: 依存関係

リスクと対策

リスク 影響 対策
図が増えすぎて読みにくくなる。 第三者が重要点を追えない。 必須図と補助図を分け、各図に目的説明を付ける。
利用者ごとに設計粒度がばらつく。 レビュー基準が不安定になる。 テンプレートと受け入れ条件を固定する。
HTML出力が崩れる。 提出物として信用されない。 出力後に自動表示チェックを行う。
権限管理の漏れ。 誤編集や情報漏えいが起きる。 閲覧、編集、レビュー、出力の権限を表で管理する。

最終判断基準: このアプリが成功したと言えるのは、出力された使用定義書を読んだ第三者が「何を、誰のために、どの範囲で、どう検証するのか」を説明できる場合である。
